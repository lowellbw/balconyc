Solar Energy Model

The energy model converts a user's address, balcony orientation, and system size into an accurate annual production estimate. The core engine is NREL's PVWatts V8 API, with balcony-specific adjustments layered on top.

Final kWh = PVWatts_Output × Shade_Factor × Urban_Soiling × Thermal_Bonus × Bifacial_Gain = PVWatts(lat, lon, capacity, tilt, azimuth, losses) × S(shadow_hours) × U(floor_level, density) × T(airflow_rating) × B(albedo, bifaciality)

1.1 PVWatts V8 API Integration

Why PVWatts? It's free, industry-standard, maintained by the US Department of Energy's National Renewable Energy Laboratory, uses the 2020 NSRDB (National Solar Radiation Database) with hourly weather data, and models all 8,760 hours of a typical meteorological year. Every commercial solar installer uses PVWatts or a derivative.

Endpoint

GET https://developer.nrel.gov/api/pvwatts/v8.json // Required: free API key from https://developer.nrel.gov/signup/ // Rate limit: 1,000 requests/hour // Output formats: .json or .xml

1.2 Balcony-Specific Parameter Mapping

The key challenge: PVWatts is designed for rooftops. Balconies differ in tilt (vertical vs. tilted), exposure (facade vs. open sky), airflow (better cooling), and soiling (urban dust). Here's how to map balcony realities to PVWatts inputs:

User InputMaps ToTransformation Logic
"My balcony faces South" azimuth = 180 Direct map: N=0, NE=45, E=90, SE=135, S=180, SW=225, W=270, NW=315
"Panels on the railing" tilt = 90 Vertical mount. If they have floor space for tilt: tilt = 70 (compromise angle)
"My balcony is about 4 ft deep × 10 ft wide" system_capacity Railing length (10 ft) fits 2× standard panels (each ~3.5 ft wide). 2 × 400W = 0.8 kW
"I'm on floor 8 of 20" losses adjustment Higher floor = less shading. Floor 8/20 → shade derating 0.90. Floor 18/20 → 0.98.
"Buildings nearby are taller" losses adjustment Adds 5-15% to losses depending on severity. See shadow model (Section 2).
"My bill is $140/month" financial model $140 ÷ $0.22/kWh ≈ 636 kWh/month. System produces ~60 kWh/month → offsets ~9.4%.

1.3 Balcony-Specific Derating Factors

PVWatts assumes a clean, unobstructed, roof-mounted system. Balconies face different real-world conditions. Apply these multipliers after PVWatts output:

FactorRooftop BaselineBalcony (Vertical)Rationale
Vertical tilt loss 1.00 (at optimal ~40° tilt) Already in PVWatts PVWatts handles tilt natively. At 90°, expect ~70-85% of optimal tilt output.
Building self-shading 1.00 0.80–0.95 Upper floors/overhangs cast morning/evening shadows on the panel. Worse for recessed balconies.
Neighboring building shade 1.00 0.70–0.95 NYC urban canyon effect. Calculated per-address in Section 2.
Urban soiling 0.97 (3%) 0.88–0.92 (8-12%) City dust, pigeon residue, pollen. Already partially in PVWatts soiling array.
Thermal benefit 1.00 1.03–1.05 Better airflow on balcony vs. roof = cooler cells = higher efficiency. ~0.35%/°C gain.
Bifacial bonus 1.00 1.05–1.15 Only if selling bifacial panels. Vertical mount + light wall behind = great rear irradiance. Use PVWatts bifaciality param instead of manual multiplier.
Implementation approach: Don't apply all derating manually. Feed the soiling array, bifaciality, and albedo directly into PVWatts parameters (it handles those internally). Then apply only the shadow derating as a post-PVWatts multiplier — because PVWatts can't model building-level shading. The shadow factor comes from Section 2's algorithm.

Simplified Derating for MVP (Before Shadow Model is Built)

// Quick derating lookup by floor level and surroundings // Use this for Phase 1 before implementing the full shadow algorithm shade_factor = { "high_floor_no_tall_neighbors": 0.95, // Penthouse, unobstructed "high_floor_some_neighbors": 0.88, // Upper third, some taller buildings "mid_floor_moderate_density": 0.82, // Middle of building, typical NYC block "mid_floor_dense_area": 0.72, // FiDi, Midtown canyon "low_floor_any": 0.60, // Floors 1-4, most of NYC "low_floor_wide_avenue": 0.75, // Low floor but on a wide avenue } // Final output: annual_kwh = pvwatts_ac_annual × shade_factor × 1.03 // thermal bonus

1.4 Financial Model

Electricity Rate

Con Edison SC-1 residential rate: ~$0.22/kWh all-in (supply + delivery). Delivery component increased ~9% in early 2026. Use $0.22 as baseline, escalate at 3%/year for payback calculations.

System Cost

800W complete kit: $1,800–$3,000 including panels, micro-inverter, mounting hardware, cables. No installation labor (DIY balcony mount). Branded kits (e.g., EcoFlow, Anker SOLIX): $2,000–2,500.

// Financial calculation annual_kwh = pvwatts_output × shade_factor × 1.03 annual_savings = annual_kwh × $0.22 monthly_savings = annual_savings / 12 bill_offset_pct = annual_kwh / (monthly_bill_$ / $0.22 × 12) × 100 // Payback with escalating rates payback_years = system_cost / annual_savings // Simple payback: ~12-20 years npv_payback_years = solve for N where: Σ(annual_savings × 1.03^i, i=0..N) = system_cost // Accounts for rate escalation // 25-year lifetime value lifetime_savings = Σ(annual_kwh × $0.22 × 1.03^i × 0.995^i, i=0..24) // 0.995^i = 0.5% annual panel degradation // 1.03^i = 3% annual electricity rate increase co2_offset_lbs = annual_kwh × 0.65 // NYC grid emission factor (lbs CO2/kWh)
What to show users: Monthly savings ($), percentage of bill offset, simple payback period, 25-year lifetime savings, and CO₂ offset. The CO₂ number is emotionally compelling — "equivalent to planting X trees per year" resonates more than raw numbers. NYC's grid is ~0.65 lbs CO₂/kWh, so 750 kWh/year = ~488 lbs CO₂ = ~4 trees/year.

Shadow Modeling

This is the hardest and most valuable piece. NYC's urban canyons mean buildings routinely block 1-6 hours of direct sunlight per day. Getting shadow analysis right is what separates a toy calculator from a trustworthy tool.

2.1 Sun Position Algorithm (SPA)

To determine if a point on a building facade is in shadow, you first need to know exactly where the sun is at any given moment. The NREL Solar Position Algorithm (SPA) is the gold standard — accurate to ±0.0003° for the years -2000 to 6000.

// Using pvlib-python (recommended implementation) import pvlib import pandas as pd # Get sun position for every hour of a typical year at a NYC location location = pvlib.location.Location( latitude=40.7484, longitude=-73.9857, tz='America/New_York', altitude=10 # meters above sea level ) # Generate hourly timestamps for a full year times = pd.date_range( '2024-01-01', '2024-12-31 23:00', freq='h', tz='America/New_York' ) # Calculate sun position using NREL SPA sun_pos = location.get_solarposition(times, method='nrel_numpy') # Returns DataFrame with columns: # apparent_zenith — angle from vertical (90° = horizon) # azimuth — compass bearing (0°=N, 90°=E, 180°=S, 270°=W) # apparent_elevation — angle above horizon (complement of zenith) # equation_of_time — correction factor (minutes) # Key NYC sun angles: # Summer solstice noon: elevation ≈ 73°, azimuth ≈ 180° # Winter solstice noon: elevation ≈ 27°, azimuth ≈ 180° # Equinox noon: elevation ≈ 50°, azimuth ≈ 180°
Why this matters for balconies: At NYC's latitude (40.7°N), the winter sun is very low in the sky (27° elevation at noon). This means buildings to the south cast extremely long shadows — a 200-foot building casts a shadow ~430 feet long at winter noon. Your shadow model must account for seasonality, or it will dramatically overestimate winter production.

2.3 Shadow Casting Algorithm

Given a target point on a building facade (the user's balcony) and the surrounding buildings, calculate what fraction of annual sunlight is blocked.

# SHADOW ANALYSIS ALGORITHM # Goal: For each hour of the year, determine if the target balcony point # is in shadow from any surrounding building. import numpy as np from pvlib import solarposition from shapely.geometry import Point, Polygon, LineString def calculate_annual_shade_factor( target_lat, target_lon, target_height_ft, surrounding_buildings, # list of {polygon, heightroof, groundelev} year=2024 ): """ Returns shade_factor (0.0–1.0) where 1.0 = no shading, 0.0 = always shaded. Also returns monthly shade factors for seasonal accuracy. """ # Step 1: Get hourly sun positions for the full year times = pd.date_range(f'{year}-01-01', f'{year}-12-31 23:00', freq='h', tz='America/New_York') sun = solarposition.get_solarposition(times, target_lat, target_lon) # Filter to daylight hours only (sun above horizon) daylight = sun[sun['apparent_elevation'] > 0] # Step 2: Convert target to local coordinate system (meters from target) target_point = Point(0, 0) # origin target_z = target_height_ft × 0.3048 # convert to meters # Step 3: For each daylight hour, check if any building casts shadow shaded_hours = 0 total_hours = len(daylight) monthly_shaded = {m: [0, 0] for m in range(1, 13)} # [shaded, total] per month for idx, row in daylight.iterrows(): sun_azimuth = np.radians(row['azimuth']) # compass bearing of sun sun_elevation = np.radians(row['apparent_elevation']) # angle above horizon month = idx.month monthly_shaded[month][1] += 1 # Direction FROM target TOWARD the sun (where light comes from) sun_dx = np.sin(sun_azimuth) # east component sun_dy = np.cos(sun_azimuth) # north component is_shaded = False for bldg in surrounding_buildings: bldg_height_m = bldg['heightroof'] × 0.3048 bldg_ground_m = bldg['groundelev'] × 0.3048 bldg_top = bldg_ground_m + bldg_height_m # Height difference between building top and our balcony delta_h = bldg_top - target_z if delta_h <= 0: continue # building is shorter than us — can't cast shadow # Maximum distance this building can cast shadow at current sun angle if sun_elevation <= 0.01: continue max_shadow_dist = delta_h / np.tan(sun_elevation) # Check if the shadow-casting ray from sun through building top # intersects our target point # Create ray from target toward sun ray_end = Point(sun_dx × max_shadow_dist, sun_dy × max_shadow_dist) sun_ray = LineString([(0, 0), (ray_end.x, ray_end.y)]) # Check if this ray intersects the building polygon bldg_poly = bldg['polygon_local'] # already in local coords (meters) if sun_ray.intersects(bldg_poly): # Check distance to intersection point intersection = sun_ray.intersection(bldg_poly) dist = target_point.distance(intersection) if dist <= max_shadow_dist: is_shaded = True break if is_shaded: shaded_hours += 1 monthly_shaded[month][0] += 1 # Step 4: Calculate factors annual_shade_factor = 1.0 - (shaded_hours / total_hours) monthly_factors = { m: 1.0 - (v[0] / max(v[1], 1)) for m, v in monthly_shaded.items() } return annual_shade_factor, monthly_factors

Key Implementation Details

Search Radius

Include buildings within 4.3 × max_neighbor_height of the target. This is the NYC CEQR (City Environmental Quality Review) standard — at minimum winter sun elevation (~27°), a building casts a shadow up to 4.3× its height. For practical purposes: 200-300 meters covers virtually all shadow-casting buildings.

Coordinate Transform

Convert lat/lon to local meters using pyproj (EPSG:2263 — NY State Plane). This gives you meters-based coordinates for distance and angle calculations. NYC footprints are in NAD83/EPSG:2263 natively.

Performance

8,760 hours × ~50-200 buildings = 400K-1.7M intersection tests. With Shapely's C-backed spatial indexing (STRtree), this runs in 2-5 seconds per address on a modern CPU. Pre-compute the STRtree once per query.

Accuracy Validation

Compare against Shadowmap.org for spot-checks. Their 3D Solar Analytics tool shows facade-level irradiance with 50km building radius and 20-year weather data. Not an API (manual comparison), but excellent for validating your algorithm.

Critical: Weight shadows by irradiance, not just hours. An hour of shadow at noon costs more energy than an hour at sunset. For accurate derating, multiply each hour's shadow status by that hour's POA irradiance from PVWatts hourly output:
# Better approach: irradiance-weighted shadow factor shade_factor = Σ(poa[h] × is_sunlit[h]) / Σ(poa[h]) for all daylight hours h # This properly weights noon shadows (high POA) more than # sunrise/sunset shadows (low POA)

2.4 Building Orientation from Footprints

If the user doesn't know which direction their balcony faces, you can infer the building's primary orientations from its footprint polygon — then ask "does your balcony face the street or the back?"

from shapely.geometry import Polygon import math def get_building_orientations(footprint_coords): """ Given a building footprint polygon, return the two primary facade orientations (the directions the longest edges face). Returns: list of azimuth angles in degrees (0-360, 0=North) """ polygon = Polygon(footprint_coords) coords = list(polygon.exterior.coords[:-1]) # exclude closing point # Calculate all edge lengths and their perpendicular directions edges = [] for i in range(len(coords)): p1 = coords[i] p2 = coords[(i + 1) % len(coords)] dx = p2[0] - p1[0] # east-west component dy = p2[1] - p1[1] # north-south component length = math.sqrt(dx**2 + dy**2) # Edge direction (along the wall) edge_angle = math.degrees(math.atan2(dx, dy)) % 360 # Facade faces PERPENDICULAR to edge — two directions (front & back) facade_angle_1 = (edge_angle + 90) % 360 facade_angle_2 = (edge_angle - 90) % 360 edges.append({ 'length': length, 'facade_angles': [facade_angle_1, facade_angle_2] }) # Sort by edge length, take the two longest (primary facades) edges.sort(key=lambda e: e['length'], reverse=True) # The longest edge's perpendicular directions = primary facade orientations primary = edges[0]['facade_angles'] # e.g., [180, 0] for a N-S facing building secondary = edges[1]['facade_angles'] # e.g., [90, 270] for E-W facades return { 'primary_facades': primary, 'secondary_facades': secondary, 'primary_edge_length': edges[0]['length'], 'confidence': 'high' if edges[0]['length'] > 1.3 × edges[1]['length'] else 'medium' } # NYC Manhattan grid: Most buildings align ~29° east of true north # So "south-facing" in Manhattan is actually ~209° azimuth # Expected accuracy: ±15° for regular buildings, ±30° for irregular lots
Manhattan's rotated grid: Manhattan's street grid runs ~29° east of true north. So a building that "faces the avenue" actually faces ~209° (SW-ish) or ~29° (NE-ish), not true south/north. This is significant for solar — a facade that appears south-facing on a map may actually face SSW. Your orientation calculator should note this for user confirmation.

Calculating the Heading (Camera → Building Direction)

import math def calculate_heading(camera_lat, camera_lon, building_lat, building_lon): """ Calculate compass bearing from camera position to building. Uses spherical trigonometry (Haversine-derived bearing formula). """ lat1 = math.radians(camera_lat) lat2 = math.radians(building_lat) d_lon = math.radians(building_lon - camera_lon) y = math.sin(d_lon) * math.cos(lat2) x = math.cos(lat1) * math.sin(lat2) - math.sin(lat1) * math.cos(lat2) * math.cos(d_lon) bearing = math.degrees(math.atan2(y, x)) heading = (bearing + 360) % 360 # normalize to 0-360 return heading # In practice: the Street View API auto-selects the nearest panorama # to your location param. The camera position is slightly offset from # the building (across the street). Heading should point AT the building. # Pitch adjustment by floor level: # Floors 1-3: pitch = 5-10° # Floors 4-8: pitch = 15-25° # Floors 9-15: pitch = 25-35° # Floors 16+: pitch = 35-50° (may not capture well from street)

4.2 Balcony Detection & Segmentation

The detection pipeline finds the balcony in the Street View image, then creates a precise pixel mask for inpainting.

1
BFA-YOLO — Balcony Bounding Box Detection

Purpose-built model for detecting building facade attachments (balconies, windows, doors, AC units). Based on YOLOv8 with three custom modules: FBSM (handles uneven object distribution), TDATH (improves small-object detection), PMESA (reduces background noise).

GitHub: github.com/CVEO/BFA-YOLO

Training data: BFA-3D dataset (multi-view facade images)

Accuracy: 91-94% mAP50 on urban building facades

Output: Bounding box [x1, y1, x2, y2] + confidence score per detected balcony

Inference: ~200ms on GPU, ~2-3s on CPU

2
SAM 3 — Pixel-Perfect Segmentation

Meta's Segment Anything Model v3 (released Nov 2025). Takes the YOLO bounding box as a "box prompt" and generates a precise pixel-level mask of the balcony — including irregular shapes, glass railings, and planters.

GitHub: github.com/facebookresearch/sam3

Key advance (v3): Promptable Concept Segmentation — can accept text prompts like "glass balcony railing" for disambiguation

API: fal.ai (fal-ai/sam-3/image) — ~$0.005-0.01/request, 1-2s latency

Output: Binary mask (white = balcony, black = preserve)

3
Depth Anything V2 — 3D Depth Map

Generates a monocular depth map from the Street View image. This tells the inpainting model about perspective — objects farther away should appear smaller, vanishing points should be respected.

GitHub: github.com/LiheYoung/Depth-Anything

Why V2 over MiDaS: AbsRel error 0.074 vs 0.127 (42% better). Superior on fine architectural details.

Output: Single-channel depth map (H×W), normalized 0-255

Used for: ControlNet depth conditioning in the inpainting step

4
Flux.2 Fill Pro — AI Inpainting

Black Forest Labs' latest inpainting model. Generates photorealistic solar panels within the masked balcony area, guided by the depth map for correct perspective.

Model: black-forest-labs/flux-fill-pro on Replicate

Cost: ~$0.025-0.05/image on fal.ai, ~$0.15/image on Replicate

Latency: 5-15 seconds depending on platform

ControlNet integration: Use controlnet_conditioning_scale = 0.75 for depth guidance — strong enough for perspective but not over-constrained

5
Post-Processing — Seamless Blending

Final compositing step to eliminate visible seams between the inpainted region and original image.

Mask feathering: Gaussian blur (σ=4-8px) on mask edges before compositing

Color matching: cv2.seamlessClone() (Poisson blending) at the boundary

Shadow generation: Project panel shadow using depth map + sun angle (45-60° typical for NYC daytime)

Optional upscaling: ESRGAN/RealESRGAN for 4K output if needed

4.3 Inpainting Prompt Engineering

Best-performing prompt (tested):
"Photorealistic modern solar panels installed on apartment balcony railing, black aluminum frame, tempered glass front with subtle blue-purple tint, realistic reflections matching ambient lighting, proper perspective, contemporary residential building, 4K detail, architectural photography" // Negative prompt (if model supports it): "cartoon, illustration, drawing, low quality, distorted perspective, floating panels, wrong scale, oversaturated"

Inpainting Parameters

ParameterValueRationale
strength0.88–0.92High enough to generate panels clearly, low enough to preserve surrounding detail
num_inference_steps25–30Quality/speed sweet spot. 20 is faster but lower quality. 50 adds latency for marginal gain.
guidance_scale7.5–8.5How closely to follow text prompt. Higher = more literal but sometimes less natural.
controlnet_conditioning_scale0.70–0.80Depth guidance strength. 0.75 is the sweet spot for architecture.
mask_blur4–8 pxPre-feather mask edges before inpainting for seamless blending.

Alternative Approach: Deterministic 3D Compositing

For cases where AI inpainting fails or for a faster MVP: Skip the AI entirely. Use OpenCV's perspective transform (cv2.findHomography) to warp a pre-rendered solar panel texture onto the detected balcony surface. Detect the balcony's four corners (from YOLO + SAM), compute the homography matrix, and warp the panel image to match the perspective. This is deterministic, instant (<1 second), and guaranteed to look correct — but less photorealistic than AI inpainting. Could serve as a fallback or "preview" while the AI version generates.

5.3 Phased Build Plan

Phase 1: Energy Estimator (3-4 weeks)

Address input with autocomplete → PLUTO building lookup → orientation pre-fill → PVWatts integration → simplified shade derating (lookup table by floor/density) → financial calculator → results dashboard with monthly production chart. This alone is a shippable product.

APIs needed: Google Places, Google Geocoding, NYC Geoclient, PLUTO, NREL PVWatts

Phase 2: AI Visualization (6-8 weeks)

Street View image fetch → BFA-YOLO balcony detection → SAM 3 segmentation → Flux.2 inpainting → post-processing → before/after slider. Runs asynchronously — energy results show immediately, visualization appears when ready.

Additional APIs: Google Street View Static, fal.ai (SAM + Flux), self-hosted YOLO + Depth

Phase 3: Precision Shadow Model (4-6 weeks)

Replace lookup-table shading with the full algorithmic shadow analysis: query neighboring building footprints + heights, run hourly sun position calculations, compute irradiance-weighted shade factor per month. This upgrades accuracy from ±20% to ±5%.

Additional data: NYC Building Footprints with heights, pvlib-python for sun position

Phase 4: Advanced Features (ongoing)

NYC LiDAR-based shadow analysis for tree canopy effects. AR/WebAR mobile experience (point phone camera at balcony, see panels in real-time). 3D panel placement configurator. Installer marketplace / lead generation. Battery storage upsell module. NYC permit guidance (DOB filings for facade-mounted systems).
